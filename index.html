RoBrain

Stop watching your AI agent repeat the same mistakes.

Open-source shared memory for teams using AI agents. Captures every decision and the alternatives your team ruled out, and flags when a new decision contradicts an old one — so your team revisits past decisions intentionally, instead of re-litigating from zero.

Claude Code Cursor Copilot

◆ Why now

Agent memory is suddenly crowded: hosted fact stores, vendor auto-memory, and markdown in the repo each solve part of the problem. Most still treat memory as what was decided — often per user, often prose — and lose what you ruled out and why, plus any signal when today’s session contradicts March’s call.

RoBrain is Postgres-backed, team-addressable, and built around structured vetoes and a full decision lifecycle. Synthesis periodically scans the whole corpus so reversals and drift surface even when no single session saw both sides. Use it alongside Auto memory if you like — RoBrain is where canonical, queryable decisions live.

◇ What makes this different

Most AI memory tools store what was decided. RoBrain also stores what was rejected and why — as structured, queryable data, not prose. Six months into a project, every architectural decision carries the full history of what your team has tried and ruled out, automatically.

Auto-memory features in modern coding agents help, but they capture per user — your teammates’ agents have no idea what you decided.

What one captured row looks like

{
  "decision": "Use Zustand for state management",
  "rationale": "Redux caused re-render issues in cart",
  "rejected": [{ "option": "Redux", "reason": "re-render perf issues in cart" }],
  "files_affected": ["src/store/cart.ts"]
}

Six months later, when your teammate’s agent suggests Redux, RoBrain surfaces the prior rejection — so the conversation starts from the full history, not from zero.

Captured

  • Architectural decisions from agent sessions
  • Rationale and rejected alternatives
  • Files in scope, timestamps, confidence

Not captured

  • Your source code or file contents
  • Secrets, passwords, or personal data
  • Anything outside your sessions with the agent

✦ What ships today

  • Passive capture — every session turn auto-recorded; no manual notes, no commands to remember.
  • rejected[] as structured data — vetoes are queryable rows in Postgres, not prose in markdown.
  • Cross-session contradiction detection — Synthesis scans the whole project on a schedule and flags reversals across sessions, even by different developers.
  • Decision lifecycle — when you change your mind, the old decision stays queryable. Ask “what did we decide in March?” three months later and get a real answer.
  • Team-shared store — one Postgres source of truth across machines and editors.
  • npx robrain explain <file> — file-scoped “why does this code exist?” from stored decisions.

◎ Architecture (OSS)

On your machine

Sensing MCP watches sessions in the background. The robrain CLI gives you review, inject, explain, and project management.

On your infrastructure

Postgres (decisions + pgvector for retrieval), the Perception API (extracts and stores decisions from session turns), and Synthesis — a batch job that periodically scans the whole corpus for contradictions, drift, and recurring entities. Runs via npx robrain synth, on a cron schedule, or both — so you’re not relying on remembering to click “scan.”

What lives in the cloud

Planning API (relevance scoring across the whole corpus) and Control MCP (proactive injection at task boundaries). Rory Plans cloud only — the OSS version runs Postgres + Perception continuously and Synthesis on whatever cadence you configure.

≋ How Synthesis catches what other tools miss

Synthesis is the job that reads the whole decision corpus. The headline pass is the contradiction scan — e.g. “Use Bun” in February vs “Migrate back to Node” in May can surface as a direct reversal across separate sessions and vocabulary, with a conflicts_with edge and both rows flagged for review.

Drift detection clusters by topic and surfaces stance shifts without an explicit reversal (think REST → GraphQL across several decisions). Entity promotion notices when a name like “Stripe” appears across many decisions and surfaces the full relationship history when any of them comes up later.

⌂ Self-hosted privacy

Your conversation turns never leave your machine. The Perception API runs locally in Docker, extraction happens on your hardware, and decisions are stored in your Postgres. Nothing is sent to Rory Plans or any other external service.

The one network dependency: extraction calls Anthropic’s Haiku API (you provide ANTHROPIC_API_KEY) and embeddings call OpenAI, Voyage, or Cohere. Those are the only outbound calls. You can swap providers or run a local model if you want full air-gapping.

⚡ Quick start

One-time setup (in the RoBrain repo)

pnpm install && pnpm build
pnpm docker:up
npx robrain install --self-hosted --repo-root "$(pwd)"

Per project

cd /path/to/your/project
npx robrain init-project

Run contradiction detection (manually or schedule via cron)

npx robrain synth

After init-project, Claude Code gets CLAUDE.md hooks for session start / turn / end — capture is automatic from there. See the README for Cursor rules and troubleshooting.
